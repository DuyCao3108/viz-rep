# Project Log: viz

Last updated: 2026-07-20 (StackBar/StackBar100 gained thresh_hold_pct + an automatic percent-label formatter in _before_plot())

## Purpose
A Python wrapper around matplotlib/numpy that automates repeatable, professional-looking BI charts with minimal per-chart boilerplate ‚Äî plus a Power BI-style dimension/measure data layer, so charts bind to declared data semantics instead of hand-shaped arrays.

## Current structure
- `src/dataset.py` ‚Äî `Dataset`/`Dimension`/`Measure` semantic data layer. **DuckDB/parquet-backed now** (was in-memory-Polars-backed) ‚Äî see Decisions.
- `src/vizzy.py` ‚Äî `Vizzy`: one method per chart family (`.bar()`/`.barh()`/`.line()`), each queries `Dataset` and returns the concrete chart object it built (e.g. `SimpleBar`). No `Artist`, no shared post-hoc styling facade.
- `src/charts/base.py` ‚Äî shape-name registry only (`register_shape`/`get_shape_class`/`_SHAPE_REGISTRY`); no shared base class anymore.
- `src/charts/` ‚Äî chart shape classes, one module per family (`bar.py`: `Bar` (plumbing) + `SimpleBar`/`SimpleBarV`/`SimpleBarH` + `StackBar`/`StackBarV`/`StackBarH` + `StackBar100`/`StackBar100V`/`StackBar100H`; `line.py`: `Line`). Each concrete class is independent ‚Äî no cross-family shared parent, no enforced method interface.
- `src/custom/` ‚Äî style/token/formatting utility modules, grouped together: `colors.py`/`fonts.py` (Economist Design System color/typography tokens) and `formatting.py` (`DataLabelFormat` Literal + `format_value()`, used by chart data-label methods). `colors.py`'s `COLOR_DICTS` is now a flat `name -> {"code": hex, "default_step": int | None}` map (no more nested `COLOR_COLLECTIONS`) ‚Äî see Decisions.
- `tests/` ‚Äî pytest suite for deterministic logic only: `test_dataset.py`, `test_colors.py`, `test_charts_registry.py`, `test_charts_bar.py`, `conftest.py` (fixtures, including `fig_ax` for chart-class tests ‚Äî forces the `Agg` backend so pytest never pops up a GUI window). Configured via `pyproject.toml` (`[project.optional-dependencies] dev`, `[tool.pytest.ini_options]`). `tests/__init__.py` (empty) makes `tests` a real package ‚Äî needed because an installed third-party `tests` package elsewhere on `sys.path` was shadowing absolute `tests.X` imports. `tests/dataset_factory.py` holds `dataset_from_df()` (materializes a generated Polars `DataFrame` to a temp parquet file and loads it as a `Dataset`, since `Dataset` no longer takes an in-memory frame) plus the random/fixed test-data generators (`_gen_data_2dim`/`_gen_data_3dim`/`_gen_orders`) shared by `conftest.py` fixtures and gallery scenarios.
- `tests/gallery/` ‚Äî visual scenario runner (`run.py` + `scenarios/{bar,line}_scenarios.py`): renders named chart scenarios to individual PNGs plus one combined contact sheet, for human review. No assertions ‚Äî a subfolder of `tests/` so pytest's default `test_*.py` collection ignores it automatically. `tests/gallery/output/` is gitignored.
- `story/pcb_card/` ‚Äî worked example wiring `Dataset` ‚Üí `Vizzy` ‚Üí rendered chart (`plot/bar_1.py`), updated to the current no-`Artist` interface.
- `archive/` ‚Äî old learning scripts and prototype analyses, not part of the library proper
- `design.txt` ‚Äî chart taxonomy: `bar` (detailed) and `line` (detailed: shapes + full config tree) both written; `special` (`waterfall`, `rank`, `sankey`, `tree`/`decomposition_tree`, `lollipop`) brainstormed at shape level. `area`/`scatter`/`pie`/`combo` not yet brainstormed.

## Decisions
- Chart taxonomy is organized as `chart type / shapes (variants) / config (common styling)`, e.g. `bar/shapes/{vertical,horizontal}/...` + `bar/config/common/{dim_axis,mes_axis,data_label,bars,spines,colors,legends}`. `line` now has the same nesting written into `design.txt` (shapes: `simple_line`, `multi_line`; config: `dim_axis`, `mes_axis`, `data_label`, `legends`, `line{smooth,size,shape,shape_at_threshold}`, `dot{size,shape,color}`).
- For `line`, stacking should not be implemented as bare stacked lines (visually unreadable) ‚Äî `stack_line`/`stack_100_line` are aliases that render as filled `area`/`stack_100_area` under the hood. Line's shape list: `simple_line`, `multi_line`, `step_line`, plus `area`, `stack_area`, `stack_100_area` as the stacking-capable siblings. Lower-priority additions: `dual_axis_line`, `line_with_markers`, `range_band_line`, `spark_line`.
- Chart classes take a matplotlib `ax` (Axes), not `plt`, as their draw target. Reason: `plt` is global pyplot state ‚Äî breaks composition of multiple charts in one figure (subplot grids/dashboards) and complicates testing. Each chart class owns drawing onto one `Axes` it's given.
- `fig` is not owned by individual chart classes ‚Äî composing multiple charts into one dashboard is constructing more than one chart object against the same `fig` with different `ax`, and the caller calls `fig.savefig(...)` themselves. No chart object has a `.save()` method.
- Added `src/dataset.py`: a Power BI-style semantic data layer ‚Äî `Dimension`/`Measure` declared once against a `Dataset` (wrapping a long-format Polars `DataFrame`), queried by name via `Dataset.query(dim=, measure=, legend=, filters=)`, returning a `QueryResult` whose `.to_tuple()` matches the `(categories, values)` / `(categories, legend_categories, values)` shape chart code expects. Backend is Polars. Scope is a single flat table only ‚Äî no cross-table joins, no multiple measures per query, no calculated/derived measures.
- **`QueryResult.categories`/`.values`/`.legend_categories` are now named wrapper dataclasses** (`ResultDimension`/`ResultMeasure`/`ResultLegend`, all with `.name`/`.value`) instead of bare `list`/`np.ndarray`. `.name` carries the dim/measure/legend's declared name through to chart classes, so `Bar.plot()`/`Line.plot()` (and their `show_data_label`'s dedup-unit subtitle) can label output with the real dim/measure name instead of a hardcoded string. `Bar`/`Line`'s public `plot(dimension, measures, legends=None)` now takes these wrapper objects directly and unwraps `.value` internally (`Bar._chart_plot`/`SimpleBar`/`StackBar` do the actual drawing on the unwrapped list/ndarray, same as before) ‚Äî `to_tuple()` still returns the 2/3-tuple, just with wrapper objects instead of bare values.
- **`Dimension.sort` (declare-time explicit category order) replaced by `Dataset.query(sort_by=...)` (query-time sort).** Reason: query-time is more PBI-like (sort is a per-chart/per-view choice, not a fixed property of the dimension) and avoids `Dimension` objects being mutated/re-declared just to change one chart's order. `sort_by: dict[Literal["dim","mes","legend"], Literal["asc","desc"]]`, default `{"dim": "asc"}` (old default behavior). `"dim"`/`"mes"` apply as a **multi-key SQL-`ORDER BY`-style sort** on categories, in dict insertion order (e.g. `{"mes": "desc", "dim": "asc"}` = sort by measure value desc, tie-break by dim name asc) ‚Äî resolved by a small `_sort_keys()` helper mapping components to Polars column names for `.sort(by=, descending=)`. `"mes"` sorts categories by aggregated measure value; when a legend is present, this sums across all legend segments per category first (`grouped.group_by(dim).agg(sum)`), matching how stacked-bar "sort by value" works in PBI/Tableau. `"legend"` is independent of `"dim"`/`"mes"` ‚Äî only reorders the legend-category (column) axis, alphabetical asc/desc by name. `sort_by` is threaded through `Vizzy.bar()`/`.barh()`/`.stack_bar()`/`.h_stack_bar()`/`.line()` the same way `filters` already was.
- **Removed `ChartShape` (cross-family shared base class) and `Artist` (post-hoc styling facade).** Reason: the shared abstract-interface pattern (`NotImplementedError` hooks forcing every shape to implement `_set_artist_color`/`_draw_data_labels`/etc.) added indirection the project wasn't getting value from yet at 2-3 shapes. Each concrete chart class now implements whatever methods it needs directly (`show_data_label`, `set_theme`, `set_horizontal_gap`, ...), accepting some duplication across classes as the tradeoff ‚Äî "each child object implements differently" rather than a forced shared contract. `Vizzy.bar()`/`.barh()`/`.line()` now return the concrete chart object itself (e.g. a `SimpleBar` instance) instead of returning `Vizzy` and requiring `.artist` indirection; chaining happens directly on the returned object (`Vizzy(fig, ax, ds).bar(dim=, measure=).show_data_label()`).
- The shape registry (`_SHAPE_REGISTRY`/`register_shape`/`get_shape_class`, still in `src/charts/base.py`) was kept even though `ChartShape` was removed ‚Äî it's independent of the base-class question, and is still how `Vizzy` resolves a shape-name string (e.g. `"simple_bar"`, `"h_simple_bar"`, `"line"`) to a class to instantiate.
- Horizontal bar resolved as a **parallel class** (`HSimpleBar`, registered `"h_simple_bar"`), not an `orientation=` flag on `Bar` ‚Äî this settles what was previously an open question. Reason: `HSimpleBar` needs `set_horizontal_gap()`, a method vertical `SimpleBar` has no use for; a flag would leave whichever orientation didn't need a given method with dead code or awkward branching. Same precedent should apply to future `StackBar`/`GroupBar`/etc. ‚Äî a parallel `H*` class each, not a flag.
- `src/styles.py` (`DefaultStyle`) **deleted entirely** ‚Äî no shared styling class. Each concrete chart class calls matplotlib directly for its own styling (e.g. `Bar.show_data_label()` calls `self.ax.bar_label(...)` directly; `Line.plot()` calls `self.ax.legend()` directly). Per-theme color cycling (`set_theme()`) is now rebuilt on `Bar` (see below); `Line.set_theme()` remains a stub.
- **`show_data_label` moved from `SimpleBar` up to `Bar`**, and extended to `show_data_label(fmt: DataLabelFormat | None = None, thresh_hold: float | None = None)`. Reason: unlike the cross-family `ChartShape` removal above, this is genuinely zero-variance logic ‚Äî `SimpleBar`/`HSimpleBar` both set `self.bars`/`self._values` identically in `plot()`, and matplotlib's `ax.bar_label()` auto-detects vertical vs. horizontal orientation from the container, so one implementation on the real (still-existing) `Bar` parent covers both via inheritance with no per-child code. `fmt` formatting logic itself lives in the new `src/custom/formatting.py` (`format_value()`) rather than inline in `bar.py`, since `design.txt`'s `line/config/data_label` will need the same K/M/B/thousand-sep/percent formatting later ‚Äî this is plain function reuse, not a forced interface, so it doesn't reintroduce the abstraction the `ChartShape` removal was avoiding.
- `src/colors.py`/`src/fonts.py` moved into a new `src/custom/` package (now `src/custom/colors.py`/`src/custom/fonts.py`), alongside the new `src/custom/formatting.py` ‚Äî groups the project's style/token/formatting utility modules under one directory instead of loose at `src/` top level.
- **`set_theme(pal, tone)` moved from per-class stubs up to `Bar`**, same zero-customization rationale as `show_data_label`, now doubly confirmed by matplotlib's `BarContainer.orientation` ‚Äî "measure axis" (y for vertical, x for horizontal) derives from `self.bars.orientation` with no per-child code. `pal: Pal` resolves via `src/custom/colors.py`'s `COLOR_THEMES`/`get_theme_colors()` (first color only ‚Äî single-series bars need exactly one); `Pal` is a `Literal` co-located in `colors.py` next to `COLOR_THEMES` so the two can't drift out of sync silently. `tone: Tone` (`Literal["minimal", "science"]`) drives small private `Bar` methods (`_hide_spines`/`_hide_measure_axis`/`_hide_ticks`/`_show_measure_axis_grid`) and stays local to `bar.py`, not a shared cross-family module ‚Äî these are `self.ax`-coupled instance methods, closer to what the `ChartShape`/`Artist` removal already rejected than to `format_value`'s pure-function reuse. Omitted params get meaningful defaults (`pal="cate-55"`, `tone="minimal"`), recovering the deleted `DefaultStyle`'s bare-axes-by-default look as an explicit opt-in-by-default call rather than forced construction-time behavior.
- Added `tests/`, split by kind: `tests/*.py` is pytest, assertion-based, for deterministic logic only (`Dataset.query()`, `colors.get_color_code`, the shape registry). `tests/gallery/` is a separate visual scenario runner with no assertions ‚Äî renders named scenarios to individual PNGs plus one contact-sheet grid image for human review, chosen over image-diff/pixel-assertion testing since chart correctness here is judged by eye, not by automated comparison. `gallery/` nests under `tests/` (not a sibling directory) specifically so pytest's default `test_*.py` collection ignores it with no extra config.

- **Bar tree restructured to `Bar ‚Üí SimpleBar ‚Üí SimpleBarV/SimpleBarH`.** The old flat `BarChart` (base) + sibling `Bar`/`BarH` concrete classes had byte-identical `plot()` bodies differing only in axis. Now: `Bar` (renamed from `BarChart`) holds plumbing + shared `show_data_label`/`set_theme`; `SimpleBar(Bar)` holds the single generic `plot()`; `SimpleBarV`/`SimpleBarH` are one-line leaves declaring `DIM_AXIS = "x"`/`"y"` and carrying the `@register_shape("simple_bar")`/`("h_simple_bar")` decorators (registry names unchanged, so `Vizzy`/callers untouched). Orientation is now a declared `DIM_AXIS` class attribute, and `_dim_axis()`/`_mes_axis()` derive from it instead of `BarContainer.orientation` ‚Äî single source of truth, known before drawing, so `set_theme()`/helpers work pre-`plot()`. The one orientation-specific line left in `plot()` is `ax.bar` vs `ax.barh`; ticks/limits resolve generically via `getattr(self.ax, f"set_{axis}...")`. Horizontal now has full parity with vertical for free (confirmed by the `_horizontal_bar_theme_defaults` gallery scenario, extended to call `set_theme()`/`show_data_label()`).
- **`StackBar`/`StackBarV`/`StackBarH` built as a sibling subtree of `Bar`, next to `SimpleBar`** (`Bar ‚Üí {SimpleBar ‚Üí SimpleBarV/SimpleBarH, StackBar ‚Üí StackBarV/StackBarH}`), registered `"stack_bar"`/`"h_stack_bar"`, exposed via new `Vizzy.stack_bar()`/`.h_stack_bar()`. `StackBar.plot(dimension, measures, legends)` requires 2D `measures` (`Dataset.query()`'s legend-pivoted shape) and a non-empty `legends`, raising `ValueError` on either violation (mirrors `SimpleBar`'s inverse `ndim != 1` check). Each legend category is drawn as its own `BarContainer` via a loop ‚Äî `ax.bar(pos, segment, bottom=cumulative, label=leg)` (vertical) / `ax.barh(..., left=cumulative, ...)` (horizontal) ‚Äî with `cumulative` running-summed after each segment; `self.bars` is therefore a **list** of containers on `StackBar`, not the single container `SimpleBar` stores. Because of that shape difference, `StackBar` **overrides** (does not inherit) `show_data_label()` (loops `self.bars`, `label_type="center"` ‚Äî segment-centered, since stacked segments don't have their own edge) and `set_theme()` (cycles `get_theme_colors(pal)[j % len(colors)]` per segment instead of `SimpleBar`'s single first-color fill ‚Äî this is the "per-category color override" the `SimpleBar` `set_theme()` decision below had flagged as moot until a legend-split shape existed). The tone-application block (`_hide_spines`/`_show_dim_spine`/`_hide_mes_axis`/`_hide_ticks`/`_show_mes_axis_grid` dispatch) was identical text between `Bar.set_theme()` and the new `StackBar.set_theme()`, so it was factored out to a new `Bar._apply_tone(tone)` private helper both call ‚Äî plain private-method reuse (same category as `format_value()`), not a reintroduction of the rejected `ChartShape` abstract-hook pattern, since nothing is forced via `NotImplementedError`.
- **Bug found and fixed while building the `StackBar` themed gallery scenario**: `plot()` calls `ax.legend()` once (needed so a legend shows at all for a legend-split chart, same as `Line.plot()`), but matplotlib's legend snapshots each handle's facecolor *at the moment `legend()` is called* ‚Äî so when `set_theme()` later recolors the bar patches to the `pal`, the legend swatches kept showing the original default-cycle colors instead of the new theme, visibly mismatching the bars. Fixed by having `StackBar.set_theme()` call `self.ax.legend()` again after recoloring, to retake the snapshot. `SimpleBar`/`Line` don't hit this (no legend at all, or legend colors never change after creation), so this only needed fixing in `StackBar.set_theme()`.
- **`src/custom/colors.py` rewritten: `COLOR_DICTS` (flat) + `COLOR_THEMES` (flexible-color-names with an optional step suffix).** The file had been left mid-broken WIP (`COLOR_DICTS` half-stubbed with `""`/`...` placeholders, `COLOR_THEMES` declared twice ‚Äî once syntactically invalid ‚Äî and two step-detection functions with no bodies); the real color data lived in a separate nested `COLOR_COLLECTIONS` (`{collection: {family: {tone: hex}}}`). Nothing outside `colors.py` referenced `COLOR_COLLECTIONS`/`COLOR_DICTS` directly (only `get_color_code`/`get_theme_colors`/`Pal`/`make_step`/`plot_theme_palette`/`plot_step_palette`, from `bar.py` and `story/pcb_card/main.py`), so it was safe to restructure. New model: `COLOR_DICTS: dict[str, dict]` is a **flat** namespace ‚Äî every color gets one distinct name and one reference hex (`{"code": hex, "default_step": int | None}`); `COLOR_COLLECTIONS`'s nested per-family ladders are gone entirely. `COLOR_THEMES` keeps its existing `dict[str, list[str]]` shape, but each entry is a "flexible-color-name": a `COLOR_DICTS` key optionally suffixed `-NN` (`NN` a multiple of 5 in `[5, 90]`) to request an HSL-lightness-derived step off that name's reference hex; no suffix falls back to the name's own `default_step` if it declared one, else its raw `code`. New `_parse_flexible_name()` (parses the optional suffix ‚Äî a trailing all-digit tail must be a multiple of 5 in range to count as a step, otherwise the whole string is treated as the base name, so names like `hcvn-red1`/`new-york` with digits/hyphens baked into the base aren't misparsed) and `_step_color()` (the HSL regen math, now parametrized directly on a hex string) replace the old dead `_detect_cur_step`/`_add_step` stubs. New `describe_theme(theme_name) -> list[tuple[str, int | None]]` reports each theme entry's `(base_name, effective_step)` for auditing what a theme actually uses. `get_color_code`/`get_theme_colors`/`make_step`/`generate_steps`/`Pal`/`plot_theme_palette`/`plot_step_palette` all kept their original signatures, so `bar.py` and `story/pcb_card/main.py` needed no changes. Also fixed data bugs found in the old `COLOR_COLLECTIONS["hcvn"]` entries (bare `{"F7374F"}` set literals instead of hex strings, missing `#` prefixes) while migrating them into `COLOR_DICTS` as `hcvn-red1`/`hcvn-red2`/`hcvn-red3`/`hcvn-dark-grey` (renamed from bare `red1`/etc. to avoid colliding with the flat brand color named `"red"`).

## Open questions
- Chart types not yet brainstormed at all: area (standalone vs. line sibling), scatter/point, pie, combo charts.
- Known gap, not yet hit in practice: with no shared base class or enforced interface across chart *families*, nothing stops `Bar.set_theme()` and `Line.set_theme()` from drifting into inconsistent behavior/signatures over time (within the `Bar` family this is now resolved ‚Äî `SimpleBar`/`HSimpleBar` share one implementation, and `StackBar`/`StackBarV`/`StackBarH` share another with a compatible but not identical signature/behavior). Acceptable tradeoff for now (see decision above); worth watching once `Line.set_theme()` gets filled in.
- `Bar.set_theme()`/`StackBar.set_theme()` have no "restore" path ‚Äî calling either once with `tone="minimal"` then again with a different `tone` on the same object won't un-hide spines/measure-axis ticks from the first call. Not yet hit in practice (nothing calls `set_theme()` twice on one object); worth a reset-before-apply if that changes.
- `StackBar`'s stray legend-color-snapshot bug (see Decisions above) suggests any future shape that both draws a legend in `plot()` *and* recolors after the fact in `set_theme()` will need the same `ax.legend()`-again fix ‚Äî worth turning into a shared helper if a third such shape shows up (`GroupBar` will, next).
- **Fixed**: the `show_data_label(..., label="Sales")` signature drift (previously listed here as an out-of-scope failure) is resolved ‚Äî `show_data_label()` no longer takes a `fmt`/`label` at all (see the format_measure() decouple in Decisions below); the test's intent (asserting subtitle text) is now served by giving the measure itself a name via its `ResultMeasure.name`/`_mes_name`, not a `label=` override. (The six `test_dataset.py` sort-order failures previously listed here are also fixed ‚Äî see Decisions, "Fixed two pre-existing `sort_by` bugs".)
- Calculated dimension columns can't reference another calculated column (only raw source columns) ‚Äî see Decisions. Revisit only if chained derived columns are actually needed.
- Cross-measure formulas and legend-present "mes" sort/`StackBar` total-label share the same pre-existing semantic approximation: both sum the *already-aggregated* per-legend-segment values rather than recomputing the underlying formula over the whole group. For non-additive measures (avg, ratio, count-distinct) this is a deliberate stacked-bar convention (the total *is* the sum of the drawn segments), not a bug, but worth remembering if a future measure type makes that reading confusing.

- **`StackBar100`/`StackBar100V`/`StackBar100H` built as a subclass of `StackBar` (not a sibling of `StackBar` under `Bar`)** ‚Äî resolves the open question the "Not yet built" section previously carried ("worth revisiting whether that's a subclass override of `plot()` or a shared normalization step"). Registered `"stack_100_bar"`/`"h_stack_100_bar"` (matching `design.txt`'s existing taxonomy naming, even though the Python class order ‚Äî `StackBar100` ‚Äî doesn't literally match the registry string order, same as `SimpleBarV`/`"simple_bar"` already not matching), exposed via new `Vizzy.stack_bar_100()`/`.h_stack_bar_100()`. Normalization is a single `_before_plot()` override: after `Bar.plot()` sets `_mes_result_vals`/`_mes_display_vals` to the raw queried array but before `_chart_plot()` draws anything, `StackBar100._before_plot()` reassigns both to a per-category percent-of-row-total array (`np.divide(..., where=row_totals != 0)`, so an all-zero category normalizes to zero segments instead of NaN). `StackBar._chart_plot()`/`show_data_label()`/`set_theme()`/`format_measure()` needed **zero changes** ‚Äî none of them assume anything about the measure array's scale, they just draw/format whatever's in `_mes_result_vals`. This is the concrete proof of the "Dataset's Dimension/Measure don't need to know about normalization" requirement: normalization is chart-class-only, and a measure's declared `fmt` (`"#"`, `"#.1"`, etc.) keeps meaning "how to render this number" ‚Äî once the number is already 0‚Äì100, formatting it is no different from formatting any other `StackBar` value. The only other override is `show_data_label(show_total=False)` (was `True` on `StackBar`) ‚Äî a 100%-stack's total is always 100, so it's not shown by default, but the param still exists if a caller wants it back.

- **`StackBar.show_data_label()` gained `thresh_hold_pct` (default `StackBar.DEFAULT_THRESHOLD_PCT = 3.5`), and `StackBar._before_plot()` gained an automatic percent-label formatter.** `thresh_hold_pct` hides a segment whose value is below X% of its own bar's total (`raw_segment / row_total * 100`), independent of the existing absolute `thresh_hold` ‚Äî both are AND-to-show checks (a label is hidden if it fails either active one; `None` disables a check). The percent formatter is a second, separate feature: `StackBar._before_plot()` now checks `_resolve_pct_fmt()` (reads the declared `Measure.fmt` via `self._mes_rf_obj.fmt` ‚Äî confirmed live but previously never consumed anywhere ‚Äî and returns it only if it's a `"%.x"` token) and, if resolved, pre-renders every raw segment value through `render_value()` into a new `_mes_display_vals` (now an `object`-dtype array of strings, not numbers), leaving `_mes_result_vals` untouched. `show_data_label()` uses a new `self._mes_pct_formatted` flag to know to use that pre-rendered string as-is instead of calling `render_value()` again. `StackBar100` overrides `_resolve_pct_fmt()` to always resolve (falling back to its own `DEFAULT_PCT_FMT = "%.1"` when no `Measure.fmt` was declared ‚Äî "default for StackBar100" per the request) and `_pct_display_scale()` to `100.0` (its values are already 0-100 post-normalization, vs. plain `StackBar`'s assumption that a `%`-fmt measure's raw value is already a 0-1 fraction ‚Äî confirmed with the user: the formatter renders each segment's own raw value as a percent, it does NOT re-derive "share of bar total" the way `thresh_hold_pct` does, even though for `StackBar100` the two happen to coincide numerically). `StackBar100._before_plot()` was reordered to normalize `_mes_result_vals` before calling `super()._before_plot()`, so the inherited percent formatter runs against the final 0-100 values instead of being stomped by normalization afterward. Bug found while wiring this up: `Bar.format_measure()` (the explicit, post-`plot()` formatting call) wasn't resetting `_mes_pct_formatted`, so an explicit `format_measure(fmt="#")` call on a `StackBar100` could get silently overridden by the always-on percent formatter ‚Äî fixed by having `format_measure()` clear the flag, since it's a deliberate override of `_mes_display_vals`. New gallery scenario `bar/vertical/stack_bar_100_pct_default` (`tests/gallery/scenarios/bar_scenarios.py`) demonstrates both features with zero extra config (no `format_measure()` call) ‚Äî visually confirmed small sector segments (~2.6-2.9% of their quarter) are hidden and every visible label renders as `"NN.N%"`.

## Decisions (cont.)
- **`Dataset` rebuilt on DuckDB reading directly from a parquet file, replacing the in-memory-Polars-DataFrame constructor.** Prototyped first in `sandbox/` (skeleton/stub code, not working ‚Äî reference for intent only, not carried over literally). `Dataset(path)` is now the *only* constructor form ‚Äî no other options, and Dataset never generates data itself (random/fixed data generation is test-only, lives in `tests/dataset_factory.py`). Every parquet column becomes a `Dimension` automatically (`DESCRIBE SELECT * FROM read_parquet(...)`, dtype `"date"` if the DuckDB type starts with `DATE`/`TIMESTAMP` else `"category"`); `Measure`/calculated `Dimension` columns are always declared afterward (`add_measure`/`set_measures`/`set_cal_col`) since DuckDB can't infer what aggregation or derived expression is wanted. `Measure.formula` is now a raw SQL aggregate expression (`"count(distinct order_id)"`, `"avg(price)"`) instead of a `column` + `AggFn` enum pair ‚Äî matches "measures are SQL count/count distinct/max" directly, no fixed agg vocabulary to extend. `filters` is now a raw SQL WHERE-clause string (was a `dict[column, value]`) ‚Äî nothing in the test suite depended on the dict form, so this was a clean swap, not a compat-shimmed one. Calculated dimension columns (`Dataset.set_cal_col(Dimension(name=, transform=))`) are spliced into a wrapping subquery (`(SELECT *, <transform> AS "<name>" FROM read_parquet(...))`) that all queries run against instead of the raw file ‚Äî known limitation: a calculated column's `transform` can only reference raw source columns, not another calculated column (they're SELECT-list siblings), acceptable until chained derived columns are actually needed.
- **Measures can reference other measures by name, DAX-style** (`Measure("pct_apple", "{num_apple_orders}/{num_orders}")`) ‚Äî confirmed in scope with the user rather than deferred. Resolved by **recursively inlining** each `{other_measure}` reference with that measure's own resolved SQL text, parenthesized (`Dataset._resolve_formula()`), until no `{}` tokens remain, producing one flat aggregate expression DuckDB evaluates in a single `GROUP BY` ‚Äî deliberately *not* multi-level/depth-aware CTEs, since arithmetic over multiple aggregate function calls in one SELECT is normal SQL and needs no intermediate materialization. Circular references (`A` formula contains `{B}`, `B`'s contains `{A}`) raise `ValueError` via a recursion-path stack.
- **DuckDB ‚Üí Polars bridge avoids pyarrow/pandas entirely**: `self._con.sql(sql).fetchnumpy()` (dict of numpy arrays, no arrow) ‚Üí `pl.DataFrame(...)` ‚Äî confirmed working in this environment, where pyarrow isn't installed and `.pl()`/`.df()` (which go through arrow) fail. Only `duckdb` was added to `pyproject.toml` dependencies.
- **`Dataset.query()`'s SQL-building is the only part that changed** ‚Äî `_sort_keys()`/`_query_2dim()`/`_query_3dim()` (multi-key sort, legend pivot into the 2D `values` array, "mes"-sort summing across legend segments) are untouched from the pre-DuckDB design, since they only ever operated on the resulting Polars `DataFrame`, not on where its rows came from. This is why the chart layer (`src/charts/bar.py`/`line.py`) needed **zero changes** ‚Äî `QueryResult`/`ResultDimension`/`ResultMeasure`/`ResultLegend` are structurally the same (`ResultMeasure` gained an additive `fmt` field carrying `Measure.fmt` through ‚Äî unused at the time, now consumed by `StackBar._resolve_pct_fmt()`'s automatic percent-label formatter, see Decisions above).
- **Fixed two pre-existing `sort_by` bugs while touching this code**: `SortOptions`/`_sort_keys()` compared against `"des"` but every call site (including the tests) passed `"desc"` ‚Äî changed the `Literal` and comparisons to `"desc"` throughout. Separately, the no-`sort_by` default was `{"mes": "desc"}` in code but documented/tested as `{"dim": "asc"}` ‚Äî changed the default to match. Both were flagged in this log previously as out-of-scope test failures; fixing them here unblocked all 6 failing `test_dataset.py` sort-order tests without changing test expectations.
- **Deduped-unit data labels**: `show_data_label(dedup_fmt_label=True, label=...)` on both `SimpleBar` and `StackBar` ‚Äî when a chart's labels all share one scale unit (K/M/B), showing the bare scaled number on every label and noting the unit once as a subtitle (e.g. "Sales (in Millions)") reads cleaner than repeating "5M"/"3M"/"10M" per label. `format_value()` in `src/custom/formatting.py` gained a `dedup_unit: bool = False` param (omits the K/M/B suffix from the rendered string when scaling; raises `ValueError` if the fmt has no scale suffix) and a new pure `unit_word(fmt)` helper (fmt's scale suffix ‚Üí "Thousands"/"Millions"/"Billions", both reusable by `Line` later per the existing `format_value` convention). `Bar` gained shared `_set_subtitle(text)` (draws once, top-left, axes-fraction coords, light grey) and `_dedup_unit_fmt(fmt)` (validates + resolves the unit word, raising with a chart-specific message) ‚Äî used identically by `SimpleBar`/`StackBar`. `StackBar._show_total_label()` also takes `dedup_fmt_label` so the total label matches the segment labels' unit-dedup state. This replaced broken WIP that had been left mid-edit in the working tree (unbound-variable bug in a since-deleted `_trunc_unit`, a syntax error in `StackBar.show_data_label`'s per-value loop, and a `dedup_fmt_label` default of `True` that would have broken every existing bare `show_data_label()` call).
- **Superseded by the raw/display value split below**: the `show_data_label(fmt=, dedup_fmt_label=)` design in the bullet above (and `_dedup_unit_fmt()`) was fully replaced ‚Äî `show_data_label()` no longer takes `fmt` at all. Kept the bullet above for history since it explains where `_set_subtitle()`/`unit_word()` originally came from.
- **Raw-vs-display value split + `Bar.format_measure()` decouple, and the `DataLabelFormat`/`DimensionFormat` token rewrite.** `Bar.plot()` now keeps two parallel copies of every dim/measure/legend array: `_..._result_vals` (raw, as queried ‚Äî never mutated, so downstream computation like sorting/totals stays exact) and `_..._display_vals` (formatted/scaled copy used only for rendering). `Bar._before_plot()` auto-applies `Dimension.fmt` (if declared) to build `_dim_display_vals` via the new `format_dimension(value, fmt)` in `src/custom/formatting.py` ‚Äî a pure function rendering date-bucket tokens (`DimensionFormat = Literal["yyyy", "yyyy-mm", "yyyy-mm-dd", "yy-mm", "yy-MM", "yy-qq"]`) off `datetime.date`/`datetime.datetime` values, raising `TypeError` for non-date values; `fmt=None` passes through `str(value)`. Dimension fmt is auto-applied every `plot()` because it's a static, declared property of the `Dimension` ‚Äî unlike measure fmt, which is an explicit, post-hoc, per-chart choice: **`Bar.format_measure(fmt, dedup_fmt=True)`** is a new public chained method (called after `plot()`, e.g. `Vizzy(...).bar(...).format_measure(fmt="#,M").show_data_label()`) that scales `_mes_display_vals` once and installs a `matplotlib.ticker.FuncFormatter` on the measure axis, so **data labels and axis ticks always render from the same fmt state and can't drift apart** ‚Äî this is the concrete answer to "formatting should apply to more than just data labels." Measure scaling is deliberately post-hoc/render-only, never a redraw: bars are always drawn from raw values (confirmed by the project owner's own `todo.txt` notes labeling this "after plot"), so `format_measure()` can be called anytime after `plot()` with no redraw needed ‚Äî the `FuncFormatter` re-renders per raw tick position on every draw. `show_data_label()` on `SimpleBar`/`StackBar` fully decoupled from formatting per the project owner's explicit choice (confirmed over a "keep fmt as a convenience passthrough" alternative): it dropped `fmt`/`dedup_fmt_label` entirely, keeping only `thresh_hold` (compares against **raw** values, per existing test expectations) and `show_total` (`StackBar`). If `format_measure()` was never called, `show_data_label()` still works (falls back to `_mes_fmt=None`/`_mes_dedup_fmt=False`, same as today's plain-whole-number default). The redundant `self._measures`/`self._dimensions`/`self._legends` (set inside `SimpleBar`/`StackBar._chart_plot()`, duplicating `Bar.plot()`'s `_mes_result_vals`/`_dim_result_vals`/`_lgd_result_vals`) were dropped as part of this pass. `DataLabelFormat` (old tokens: `"whole-k"`/`"whole-m"`/`"whole-b"`/`*-thsep`/`"whole-thsep"`/`"%.2"`/`"%.3"`) was fully retired and replaced with a new token set (`"#"`, `"#,"`, `"#,K"`, `"#,M"`, `"#,B"`, `"#."`, `"#.1"`/`"#.2"`/`"#.3"`, `"%.1"`/`"%.2"`/`"%.3"`) ‚Äî every `#,K`/`#,M`/`#,B` always implies thousand-sep (the old no-thsep scaled variant has no equivalent; minor cosmetic behavior change, not a regression). The scale-computation/string-rendering split (`scale_factor(fmt)`, `unit_word(fmt)`, `render_value(already_scaled_value, fmt, dedup_unit)`, `format_value(raw_value, fmt, dedup_unit)` as the one-shot scale+render composition) replaces the old monolithic `format_value()` so `format_measure()`'s precomputed array and the tick `FuncFormatter`'s per-draw closure share one source of truth instead of duplicating scale/suffix logic.
