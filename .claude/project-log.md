# Project Log: viz

Last updated: 2026-07-18

## Purpose
A Python wrapper around matplotlib/numpy that automates repeatable, professional-looking BI charts with minimal per-chart boilerplate — plus a Power BI-style dimension/measure data layer, so charts bind to declared data semantics instead of hand-shaped arrays.

## Current structure
- `src/dataset.py` — `Dataset`/`Dimension`/`Measure` semantic data layer (Polars-backed)
- `src/vizzy.py` — `Vizzy`: one method per chart family (`.bar()`/`.barh()`/`.line()`), each queries `Dataset` and returns the concrete chart object it built (e.g. `SimpleBar`). No `Artist`, no shared post-hoc styling facade.
- `src/charts/base.py` — shape-name registry only (`register_shape`/`get_shape_class`/`_SHAPE_REGISTRY`); no shared base class anymore.
- `src/charts/` — chart shape classes, one module per family (`bar.py`: `Bar` (plumbing) + `SimpleBar`/`SimpleBarV`/`SimpleBarH` + `StackBar`/`StackBarV`/`StackBarH`; `line.py`: `Line`). Each concrete class is independent — no cross-family shared parent, no enforced method interface.
- `src/custom/` — style/token/formatting utility modules, grouped together: `colors.py`/`fonts.py` (Economist Design System color/typography tokens) and `formatting.py` (`DataLabelFormat` Literal + `format_value()`, used by chart data-label methods)
- `tests/` — pytest suite for deterministic logic only: `test_dataset.py`, `test_colors.py`, `test_charts_registry.py`, `test_charts_bar.py`, `conftest.py` (fixtures, including `fig_ax` for chart-class tests — forces the `Agg` backend so pytest never pops up a GUI window). Configured via `pyproject.toml` (`[project.optional-dependencies] dev`, `[tool.pytest.ini_options]`).
- `tests/gallery/` — visual scenario runner (`run.py` + `scenarios/{bar,line}_scenarios.py`): renders named chart scenarios to individual PNGs plus one combined contact sheet, for human review. No assertions — a subfolder of `tests/` so pytest's default `test_*.py` collection ignores it automatically. `tests/gallery/output/` is gitignored.
- `story/pcb_card/` — worked example wiring `Dataset` → `Vizzy` → rendered chart (`plot/bar_1.py`), updated to the current no-`Artist` interface.
- `archive/` — old learning scripts and prototype analyses, not part of the library proper
- `design.txt` — chart taxonomy: `bar` (detailed) and `line` (detailed: shapes + full config tree) both written; `special` (`waterfall`, `rank`, `sankey`, `tree`/`decomposition_tree`, `lollipop`) brainstormed at shape level. `area`/`scatter`/`pie`/`combo` not yet brainstormed.

## Decisions
- Chart taxonomy is organized as `chart type / shapes (variants) / config (common styling)`, e.g. `bar/shapes/{vertical,horizontal}/...` + `bar/config/common/{dim_axis,mes_axis,data_label,bars,spines,colors,legends}`. `line` now has the same nesting written into `design.txt` (shapes: `simple_line`, `multi_line`; config: `dim_axis`, `mes_axis`, `data_label`, `legends`, `line{smooth,size,shape,shape_at_threshold}`, `dot{size,shape,color}`).
- For `line`, stacking should not be implemented as bare stacked lines (visually unreadable) — `stack_line`/`stack_100_line` are aliases that render as filled `area`/`stack_100_area` under the hood. Line's shape list: `simple_line`, `multi_line`, `step_line`, plus `area`, `stack_area`, `stack_100_area` as the stacking-capable siblings. Lower-priority additions: `dual_axis_line`, `line_with_markers`, `range_band_line`, `spark_line`.
- Chart classes take a matplotlib `ax` (Axes), not `plt`, as their draw target. Reason: `plt` is global pyplot state — breaks composition of multiple charts in one figure (subplot grids/dashboards) and complicates testing. Each chart class owns drawing onto one `Axes` it's given.
- `fig` is not owned by individual chart classes — composing multiple charts into one dashboard is constructing more than one chart object against the same `fig` with different `ax`, and the caller calls `fig.savefig(...)` themselves. No chart object has a `.save()` method.
- Added `src/dataset.py`: a Power BI-style semantic data layer — `Dimension`/`Measure` declared once against a `Dataset` (wrapping a long-format Polars `DataFrame`), queried by name via `Dataset.query(dim=, measure=, legend=, filters=)`, returning a `QueryResult` whose `.to_tuple()` matches the `(categories, values)` / `(categories, legend_categories, values)` shape chart code expects. Backend is Polars. Scope is a single flat table only — no cross-table joins, no multiple measures per query, no calculated/derived measures.
- **Removed `ChartShape` (cross-family shared base class) and `Artist` (post-hoc styling facade).** Reason: the shared abstract-interface pattern (`NotImplementedError` hooks forcing every shape to implement `_set_artist_color`/`_draw_data_labels`/etc.) added indirection the project wasn't getting value from yet at 2-3 shapes. Each concrete chart class now implements whatever methods it needs directly (`show_data_label`, `set_theme`, `set_horizontal_gap`, ...), accepting some duplication across classes as the tradeoff — "each child object implements differently" rather than a forced shared contract. `Vizzy.bar()`/`.barh()`/`.line()` now return the concrete chart object itself (e.g. a `SimpleBar` instance) instead of returning `Vizzy` and requiring `.artist` indirection; chaining happens directly on the returned object (`Vizzy(fig, ax, ds).bar(dim=, measure=).show_data_label()`).
- The shape registry (`_SHAPE_REGISTRY`/`register_shape`/`get_shape_class`, still in `src/charts/base.py`) was kept even though `ChartShape` was removed — it's independent of the base-class question, and is still how `Vizzy` resolves a shape-name string (e.g. `"simple_bar"`, `"h_simple_bar"`, `"line"`) to a class to instantiate.
- Horizontal bar resolved as a **parallel class** (`HSimpleBar`, registered `"h_simple_bar"`), not an `orientation=` flag on `Bar` — this settles what was previously an open question. Reason: `HSimpleBar` needs `set_horizontal_gap()`, a method vertical `SimpleBar` has no use for; a flag would leave whichever orientation didn't need a given method with dead code or awkward branching. Same precedent should apply to future `StackBar`/`GroupBar`/etc. — a parallel `H*` class each, not a flag.
- `src/styles.py` (`DefaultStyle`) **deleted entirely** — no shared styling class. Each concrete chart class calls matplotlib directly for its own styling (e.g. `Bar.show_data_label()` calls `self.ax.bar_label(...)` directly; `Line.plot()` calls `self.ax.legend()` directly). Per-theme color cycling (`set_theme()`) is now rebuilt on `Bar` (see below); `Line.set_theme()` remains a stub.
- **`show_data_label` moved from `SimpleBar` up to `Bar`**, and extended to `show_data_label(fmt: DataLabelFormat | None = None, thresh_hold: float | None = None)`. Reason: unlike the cross-family `ChartShape` removal above, this is genuinely zero-variance logic — `SimpleBar`/`HSimpleBar` both set `self.bars`/`self._values` identically in `plot()`, and matplotlib's `ax.bar_label()` auto-detects vertical vs. horizontal orientation from the container, so one implementation on the real (still-existing) `Bar` parent covers both via inheritance with no per-child code. `fmt` formatting logic itself lives in the new `src/custom/formatting.py` (`format_value()`) rather than inline in `bar.py`, since `design.txt`'s `line/config/data_label` will need the same K/M/B/thousand-sep/percent formatting later — this is plain function reuse, not a forced interface, so it doesn't reintroduce the abstraction the `ChartShape` removal was avoiding.
- `src/colors.py`/`src/fonts.py` moved into a new `src/custom/` package (now `src/custom/colors.py`/`src/custom/fonts.py`), alongside the new `src/custom/formatting.py` — groups the project's style/token/formatting utility modules under one directory instead of loose at `src/` top level.
- **`set_theme(pal, tone)` moved from per-class stubs up to `Bar`**, same zero-customization rationale as `show_data_label`, now doubly confirmed by matplotlib's `BarContainer.orientation` — "measure axis" (y for vertical, x for horizontal) derives from `self.bars.orientation` with no per-child code. `pal: Pal` resolves via `src/custom/colors.py`'s `COLOR_THEMES`/`get_theme_colors()` (first color only — single-series bars need exactly one); `Pal` is a `Literal` co-located in `colors.py` next to `COLOR_THEMES` so the two can't drift out of sync silently. `tone: Tone` (`Literal["minimal", "science"]`) drives small private `Bar` methods (`_hide_spines`/`_hide_measure_axis`/`_hide_ticks`/`_show_measure_axis_grid`) and stays local to `bar.py`, not a shared cross-family module — these are `self.ax`-coupled instance methods, closer to what the `ChartShape`/`Artist` removal already rejected than to `format_value`'s pure-function reuse. Omitted params get meaningful defaults (`pal="cate-55"`, `tone="minimal"`), recovering the deleted `DefaultStyle`'s bare-axes-by-default look as an explicit opt-in-by-default call rather than forced construction-time behavior.
- Added `tests/`, split by kind: `tests/*.py` is pytest, assertion-based, for deterministic logic only (`Dataset.query()`, `colors.get_color_code`, the shape registry). `tests/gallery/` is a separate visual scenario runner with no assertions — renders named scenarios to individual PNGs plus one contact-sheet grid image for human review, chosen over image-diff/pixel-assertion testing since chart correctness here is judged by eye, not by automated comparison. `gallery/` nests under `tests/` (not a sibling directory) specifically so pytest's default `test_*.py` collection ignores it with no extra config.

- **Bar tree restructured to `Bar → SimpleBar → SimpleBarV/SimpleBarH`.** The old flat `BarChart` (base) + sibling `Bar`/`BarH` concrete classes had byte-identical `plot()` bodies differing only in axis. Now: `Bar` (renamed from `BarChart`) holds plumbing + shared `show_data_label`/`set_theme`; `SimpleBar(Bar)` holds the single generic `plot()`; `SimpleBarV`/`SimpleBarH` are one-line leaves declaring `DIM_AXIS = "x"`/`"y"` and carrying the `@register_shape("simple_bar")`/`("h_simple_bar")` decorators (registry names unchanged, so `Vizzy`/callers untouched). Orientation is now a declared `DIM_AXIS` class attribute, and `_dim_axis()`/`_mes_axis()` derive from it instead of `BarContainer.orientation` — single source of truth, known before drawing, so `set_theme()`/helpers work pre-`plot()`. The one orientation-specific line left in `plot()` is `ax.bar` vs `ax.barh`; ticks/limits resolve generically via `getattr(self.ax, f"set_{axis}...")`. Horizontal now has full parity with vertical for free (confirmed by the `_horizontal_bar_theme_defaults` gallery scenario, extended to call `set_theme()`/`show_data_label()`).
- **`StackBar`/`StackBarV`/`StackBarH` built as a sibling subtree of `Bar`, next to `SimpleBar`** (`Bar → {SimpleBar → SimpleBarV/SimpleBarH, StackBar → StackBarV/StackBarH}`), registered `"stack_bar"`/`"h_stack_bar"`, exposed via new `Vizzy.stack_bar()`/`.h_stack_bar()`. `StackBar.plot(dimension, measures, legends)` requires 2D `measures` (`Dataset.query()`'s legend-pivoted shape) and a non-empty `legends`, raising `ValueError` on either violation (mirrors `SimpleBar`'s inverse `ndim != 1` check). Each legend category is drawn as its own `BarContainer` via a loop — `ax.bar(pos, segment, bottom=cumulative, label=leg)` (vertical) / `ax.barh(..., left=cumulative, ...)` (horizontal) — with `cumulative` running-summed after each segment; `self.bars` is therefore a **list** of containers on `StackBar`, not the single container `SimpleBar` stores. Because of that shape difference, `StackBar` **overrides** (does not inherit) `show_data_label()` (loops `self.bars`, `label_type="center"` — segment-centered, since stacked segments don't have their own edge) and `set_theme()` (cycles `get_theme_colors(pal)[j % len(colors)]` per segment instead of `SimpleBar`'s single first-color fill — this is the "per-category color override" the `SimpleBar` `set_theme()` decision below had flagged as moot until a legend-split shape existed). The tone-application block (`_hide_spines`/`_show_dim_spine`/`_hide_mes_axis`/`_hide_ticks`/`_show_mes_axis_grid` dispatch) was identical text between `Bar.set_theme()` and the new `StackBar.set_theme()`, so it was factored out to a new `Bar._apply_tone(tone)` private helper both call — plain private-method reuse (same category as `format_value()`), not a reintroduction of the rejected `ChartShape` abstract-hook pattern, since nothing is forced via `NotImplementedError`.
- **Bug found and fixed while building the `StackBar` themed gallery scenario**: `plot()` calls `ax.legend()` once (needed so a legend shows at all for a legend-split chart, same as `Line.plot()`), but matplotlib's legend snapshots each handle's facecolor *at the moment `legend()` is called* — so when `set_theme()` later recolors the bar patches to the `pal`, the legend swatches kept showing the original default-cycle colors instead of the new theme, visibly mismatching the bars. Fixed by having `StackBar.set_theme()` call `self.ax.legend()` again after recoloring, to retake the snapshot. `SimpleBar`/`Line` don't hit this (no legend at all, or legend colors never change after creation), so this only needed fixing in `StackBar.set_theme()`.

## Not yet built (chart layer extension points)
- **Stubs with no body yet** (immediate next-build items): `Line.show_data_label()`/`set_smooth()`/`set_line_style()`/`set_theme()`. (The horizontal simple bar `SimpleBarH` is now fully built and shares `plot()` with `SimpleBarV` — no longer a stub; `set_horizontal_gap()` was never needed once `plot()` was shared.)
- **Bar**: `Stack100Bar`, `GroupBar`, `GroupStackBar`, `DiverseBar` — subclass `Bar` in `charts/bar.py` — plus each one's `H*` parallel-class sibling (see horizontal-bar decision above; no longer an open flag-vs-class question). (`StackBar`/`StackBarV`/`StackBarH` are now built — see Decisions above. `Stack100Bar` should be a thin variant reusing most of `StackBar.plot()`'s loop but normalizing each category's segments to sum to 100 first — worth revisiting whether that's a subclass override of `plot()` or a shared normalization step once it's built, rather than deciding now.)
- **Line**: `stack_line`/`stack_100_line` (must render as filled area, never bare stacked lines), `step_line`, `dual_axis_line`, `line_with_markers`, `range_band_line`, `spark_line`.
- **Special charts**: `Waterfall`, `Rank`, `Sankey`, `DecompositionTree`, `Lollipop` — shape-level taxonomy now in `design.txt`, nothing built yet.
- Combining two chart types on one `ax` (e.g. dual-axis bar+line) — not designed for yet.
- `Dimension(dtype="date")` exists in `dataset.py` but nothing in `charts/` does date-aware axis formatting yet.
- `line/config/data_label`'s `show_threshold`/`show_exact` and `legends_on_line` config knobs from `design.txt` — `Bar`'s equivalent (`show_data_label(fmt=, thresh_hold=)`) is now built; `Line.show_data_label()` is still a stub and should reuse `src/custom/formatting.py` rather than reimplementing.
- Power BI-style hand-drawn horizontal token legend (`build_report.py`'s `hlegend`) as an upgrade over the plain `ax.legend()` call each chart class makes directly now.
- Per-category (not just per-series) color overrides on `SimpleBar` (today `set_theme()` recolors every bar uniformly to `pal`'s first color, since `SimpleBar`/`HSimpleBar` are always single-series) — resolved for `StackBar` (see Decisions above); still moot for `SimpleBar` itself until a shape needs mixed per-bar coloring within one series.

## Open questions
- Chart types not yet brainstormed at all: area (standalone vs. line sibling), scatter/point, pie, combo charts.
- Known gap, not yet hit in practice: with no shared base class or enforced interface across chart *families*, nothing stops `Bar.set_theme()` and `Line.set_theme()` from drifting into inconsistent behavior/signatures over time (within the `Bar` family this is now resolved — `SimpleBar`/`HSimpleBar` share one implementation, and `StackBar`/`StackBarV`/`StackBarH` share another with a compatible but not identical signature/behavior). Acceptable tradeoff for now (see decision above); worth watching once `Line.set_theme()` gets filled in.
- `Bar.set_theme()`/`StackBar.set_theme()` have no "restore" path — calling either once with `tone="minimal"` then again with a different `tone` on the same object won't un-hide spines/measure-axis ticks from the first call. Not yet hit in practice (nothing calls `set_theme()` twice on one object); worth a reset-before-apply if that changes.
- `StackBar`'s stray legend-color-snapshot bug (see Decisions above) suggests any future shape that both draws a legend in `plot()` *and* recolors after the fact in `set_theme()` will need the same `ax.legend()`-again fix — worth turning into a shared helper if a third such shape shows up (`GroupBar` will, next).
